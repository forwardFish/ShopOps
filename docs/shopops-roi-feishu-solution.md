# ShopOps 多平台投流 ROI 与真实收益核算方案

> 用途：这份文档用于向 ChatGPT 或其他开发代理说明 ShopOps 当前项目背景、业务痛点、飞书多维表格数据库约束，以及下一步要设计/实现的“每投流 1000 元到底赚不赚钱”核算方案。

## 1. 一句话目标

ShopOps 要解决商家最关心的问题：

> 我在多个平台投流 1000 元，扣掉退款、达人佣金等后，到底带来了多少真实成交、多少净收益，是赚钱还是亏钱？

系统不能只展示平台后台给出的 ROI，因为平台后台通常只看广告消耗和成交额，容易忽略：

- 多平台订单口径不一致；
- 多个广告平台、多个店铺、多个投流账户；
- 抖音/达人带货产生的预估佣金、结算佣金、技术服务费；
- 用户下单后过几天退款，导致当天看起来赚钱，几天后实际亏损；
- 推广费、达人佣金、退款发生时间不同；
- 部分数据采集失败时不能把失败写成 0，避免错误判断。

最终要给商家一个清晰结论：

- 今天每投流 1000 元，预计带来多少净成交？
- 退款回补后，每投流 1000 元最终带来多少净成交？
- 扣掉达人佣金后，每投流 1000 元剩下多少已知经营贡献？
- 哪个平台赚钱，哪个平台亏钱？
- 当前数据是“实时预估”“观察中”还是“已结算”？

## 2. ShopOps 当前项目背景

项目路径：

```text
D:\lyh\agent\agent-frame\ShopOps
```

ShopOps 当前是一个 Python 后端任务型项目，用于采集电商经营数据、计算指标、写入飞书多维表格，并生成监控、告警和日报。

当前已有能力包括：

- 订单采集：
  - 淘宝/天猫：千牛 PC 页面采集或淘宝 API 边界；
  - 拼多多：平台订单 API 边界；
  - 抖店/抖音：平台订单 API 边界；
  - 视频号/微信小店：平台订单 API 边界；
  - 聚水潭：统一订单 API 路径。
- 推广花费采集：
  - 当前 MVP 主要从千牛 PC 推广中心读取推广花费；
  - 项目中已有 `promotion_snapshot`、`ad_data` 示例结构，可扩展到更多平台。
- 达人佣金：
  - 已有聚水潭抖音达人佣金采集器；
  - 已有飞书表：`douyin_influencer_commission`，中文表名意图为“抖音达人佣金明细表”。
- 指标计算：
  - 当前已有 `monitor_snapshot` 实时监控快照；
  - 当前已有 `metrics_10min` 十分钟增量指标；
  - 当前已有 ROI、CAC、日报、告警、任务日志。
- 存储：
  - 当前数据库就是飞书多维表格；
  - 本地开发有 `local_feishu.json` 作为飞书多维表格替身；
  - 正式运行时通过飞书 OpenAPI 写入真实多维表格；
  - 所有核心记录依赖 `unique_key` 做 upsert，避免重复写入。

非常重要：

> 当前不要默认使用 MySQL、PostgreSQL、ClickHouse 等数据库。ShopOps 当前数据库是飞书多维表格，方案必须围绕飞书多维表格设计。

## 3. 业务问题拆解

商家说的“投入产出比”其实有多种口径，必须拆清楚，否则会越看越糊涂。

### 3.1 平台后台 ROI

平台后台 ROI 通常是：

```text
平台后台 ROI = 平台归因成交额 / 平台广告消耗
```

问题：

- 平台可能用自己的归因窗口；
- 可能统计下单金额，不一定扣退款；
- 可能不扣达人佣金；
- 不同平台口径不一致；
- 多平台合计时容易重复或漏算。

这个指标可以保留，但不能作为最终经营判断。

### 3.2 实时预估 ROI

实时预估 ROI 用来回答：

> 截止现在，今天投的钱大概带来了多少成交？

建议公式：

```text
实时预估 ROI = 今日净成交额预估 / 今日广告消耗

今日净成交额预估 = 今日已支付成交额 - 今日已发生退款额
```

注意：今天还没退款的订单未来可能退款，所以这个指标必须标记为“预估”。

### 3.3 含达人佣金 ROI

对于抖音/达人带货，真实投放成本不只有广告费，还包括达人佣金。

建议公式：

```text
含佣金 ROI = 净成交额 / (广告消耗 + 达人佣金)
```

其中达人佣金分两种：

```text
预估达人佣金 = 平台或聚水潭返回的 estimated_commission
结算达人佣金 = 平台或聚水潭返回的 settled_commission / real_commission
```

看实时数据时用预估佣金；看最终结果时用结算佣金。

### 3.4 每投流 1000 元产出

商家最容易理解的展示方式：

```text
每投 1000 元净成交 = 净成交额 / 广告消耗 * 1000
```

如果要把达人佣金也算进投入：

```text
每投入 1000 元净成交（含佣金） = 净成交额 / (广告消耗 + 达人佣金) * 1000
```

如果要看“已知经营贡献”，但还没有商品成本、运费、平台扣点，则只能先做已知贡献：

```text
已知经营贡献 = 净成交额 - 广告消耗 - 达人佣金

每投 1000 元已知经营贡献 = 已知经营贡献 / 广告消耗 * 1000
```

如果后续拿到商品成本、运费、平台扣点，再升级为毛利 ROI：

```text
毛利 = 净成交额 - 商品成本 - 运费 - 平台扣点 - 达人佣金
投流净利润 = 毛利 - 广告消耗
```

当前方案不要假装已经有商品成本数据。

## 4. 退款滞后问题怎么解决

退款是 ROI 混乱的核心原因。解决思路是：不要只保存“今天订单汇总”，必须保存订单和退款的生命周期。

### 4.1 保留两种日期口径

同一笔退款要支持两个视角：

1. 按支付日回冲：回答“那天投的钱最终赚没赚？”
2. 按退款发生日记账：回答“今天现金流被退款影响多少？”

例子：

```text
6 月 1 日投流产生订单 500 元
6 月 5 日客户退款 500 元
```

按支付日回冲：

```text
6 月 1 日最终净成交额减少 500 元
```

按退款发生日记账：

```text
6 月 5 日退款影响 500 元
```

商家判断投放效果时，核心看“按支付日回冲”的最终 ROI；财务现金流看“按退款发生日”的退款影响。

### 4.2 设置成熟期状态

每个平台退款周期不同，建议给每个平台配置观察期。

建议默认：

| 状态 | 含义 | 示例 |
|---|---|---|
| 实时预估 | 今天或近 1 天订单，退款未充分暴露 | T+0 |
| 观察中 | 订单仍可能退款 | T+1 到 T+7 |
| 基本稳定 | 大部分退款已暴露 | T+7 |
| 已结算 | 超过平台配置的最终观察期 | T+15 或 T+30 |

飞书汇总表里必须有 `结算成熟度` 字段，不允许把 T+0 ROI 和 T+15 ROI 混为一谈。

### 4.3 采集策略

每次任务不要只拉今天订单，还要回看最近 N 天的订单变更。

建议：

```text
每 10 分钟：
1. 拉取今天订单和推广花费；
2. 拉取最近 15 到 30 天的订单变更状态；
3. 拉取最近 15 到 30 天的退款/售后状态；
4. 更新订单表里的累计退款额、净成交额、订单最终状态；
5. 重算被影响日期的 ROI 汇总。
```

这样 6 月 5 日发生退款时，可以自动回改 6 月 1 日的最终 ROI。

## 5. 飞书多维表格数据库设计

飞书多维表格是当前数据库，因此表结构要兼顾：

- 人能直接看懂；
- Python 程序能 upsert；
- 字段不要过度复杂；
- 每张表都要有 `unique_key`；
- 采集失败时写状态，不写 0 污染指标；
- 原始数据保留在 `原始数据` 字段，方便追溯。

## 6. 指标计算规则

### 6.1 基础金额

```text
已支付成交额 = sum(order_lifecycle.支付金额)
已成功退款额 = sum(refund_event.退款金额 where 退款状态 = 退款成功)
净成交额 = 已支付成交额 - 已成功退款额
广告消耗 = sum(ad_cost_ledger.推广花费)
达人佣金 = 优先使用结算佣金；未结算时使用预估佣金
已知总投入 = 广告消耗 + 达人佣金
```

### 6.2 ROI

```text
真实ROI_仅广告 = 净成交额 / 广告消耗
真实ROI_含佣金 = 净成交额 / (广告消耗 + 达人佣金)
```

当分母为 0、空或采集失败时：

```text
ROI = null
数据状态 = failed / missing_cost / partial
```

不要写 0。

### 6.3 每投 1000 元

```text
每投1000净成交_仅广告 = 净成交额 / 广告消耗 * 1000
每投1000净成交_含佣金 = 净成交额 / (广告消耗 + 达人佣金) * 1000
每投1000已知经营贡献 = (净成交额 - 广告消耗 - 达人佣金) / 广告消耗 * 1000
```

解释方式：

```text
每投1000净成交_仅广告 = 6800
代表广告费每花 1000 元，扣退款后带来 6800 元净成交。

每投1000已知经营贡献 = 1200
代表广告费每花 1000 元，扣广告费和达人佣金后，当前已知还剩 1200 元经营贡献。
```

### 6.4 平台对比

每个平台独立计算：

```text
平台 = 天猫 / 抖音 / 拼多多 / 视频号
```

再合计全平台：

```text
全平台广告消耗 = sum(各平台广告消耗)
全平台净成交额 = sum(各平台净成交额)
全平台达人佣金 = sum(各平台达人佣金)
全平台真实ROI = 全平台净成交额 / 全平台已知总投入
```

不要把各平台 ROI 简单平均，必须用金额加权。

## 7. 多平台归因策略

### 7.1 第一阶段：平台日级归因

先做最稳的版本：

```text
同一平台、同一店铺、同一统计日期：
广告消耗和订单净成交做日级汇总。
```

这可以回答：

> 今天抖音投了多少钱，抖音店铺扣退款后带来多少净成交？

优点：

- 飞书多维表格容易实现；
- 不要求每个平台都有订单级广告归因；
- 适合老板看总账。

缺点：

- 不能精确判断某个计划、某个达人、某个素材赚钱没赚钱。

### 7.2 第二阶段：商品/计划级归因

如果广告消耗里有商品 ID、计划 ID，订单里也有商品 ID，可以进一步做：

```text
平台 + 店铺 + 商品ID + 日期
```

或：

```text
平台 + 店铺 + 广告计划ID + 商品ID + 日期
```

### 7.3 第三阶段：订单级归因

如果平台或聚水潭能返回订单来源、达人、广告计划，则建立 `roi_attribution_detail`。

建议字段：

| 字段名 | 说明 |
|---|---|
| `unique_key` | `平台_订单号_归因来源` |
| `订单号` | 订单 |
| `平台` | 平台 |
| `归因方式` | 平台回传、达人订单、商品日级分摊、手工规则 |
| `广告计划ID` | 可选 |
| `达人ID` | 可选 |
| `归因广告消耗` | 分摊到订单的广告成本 |
| `归因达人佣金` | 分摊到订单的达人佣金 |
| `净成交额` | 订单净成交 |
| `订单ROI` | 订单净成交 / 归因投入 |
| `置信度` | high、medium、low |
| `说明` | 归因口径 |

不能精确归因时必须标记置信度，不能假装精确。

## 8. 告警和老板可读结论

### 8.1 必要告警

建议新增或强化这些告警：

| 告警 | 条件 |
|---|---|
| ROI 低 | `真实ROI_含佣金 < 配置阈值` |
| 每投1000亏损 | `每投1000已知经营贡献 < 0` |
| 退款回冲异常 | 某天 ROI 因退款下降超过阈值 |
| 达人佣金异常 | 佣金率或佣金金额高于阈值 |
| 平台后台 ROI 与真实 ROI 差异大 | `abs(平台显示ROI - 真实ROI) > 阈值` |
| 数据采集失败 | 订单、投流、佣金、退款任一关键来源失败 |
| 数据不成熟 | T+0 数据不能作为最终结论 |

### 8.2 日报文案建议

日报不要只写数字，要直接写结论。

示例：

```text
今日全平台广告消耗 10,000 元，扣退款后净成交额 62,000 元。
按广告费口径，每投 1000 元预计带来 6200 元净成交。
扣除当前可见达人佣金 4,800 元后，每投 1000 元预计已知经营贡献为 4720 元。

其中抖音 ROI 受达人佣金影响最大，平台后台 ROI 为 6.8，真实含佣金 ROI 为 5.2。
拼多多今日退款回冲较高，导致 6 月 1 日最终 ROI 从 4.1 降至 3.2。

当前结论为“实时预估”，近 7 天订单仍可能退款，请以 T+15 已结算 ROI 作为最终投放复盘。
```

## 11. 最重要的产品原则

1. 飞书多维表格是当前数据库，不是临时导出结果。
2. ROI 必须分“实时预估”和“最终结算”。
3. 退款必须能回冲原支付日，否则历史投放复盘是错的。
4. 达人佣金必须进入投入成本，否则抖音 ROI 会虚高。
5. 平台后台 ROI 只能作参考，真实经营 ROI 要用 ShopOps 自己统一口径计算。
6. 每投 1000 元产出要用老板能理解的话展示。
7. 采集失败不能写 0，必须写 null 和数据状态。
8. 无法精确归因时，要标明归因方式和置信度。
